Deployment Stability Notes (2026-02-21)

•       GHCR image verify uchun repository nomi lowercase bo'lishi shart; sha tag short format (`sha-xxxxxxx`) bilan ishlatiladi.

•       Web/Platform Docker runtime command standart: `bun run --cwd apps/web start`; API runtime command: `bun run --cwd apps/api start:prod`; worker (if API image reused): `bun run --cwd apps/api start:worker:prod`.

•       Build artifact check majburiy: web `.next/BUILD_ID`, api `dist/main(.js)`; Dokploy `Run Command` noto'g'ri override qilinmasin (image CMD ishlasin). Istisno: worker service API image'ni boshqa command bilan ishlatsa aniq override kiriting.

•       API ValidationPipe ishlatilganda `class-validator` va `class-transformer` dependencies ichida bo'lishi kerak.

•       Sentry test endpoint vaqtinchalik: event tekshiruvdan so'ng production'dan olib tashlanadi.

  DOKPLOY DEPLOY RUNBOOK (talimy.space)

  1) Project: `talimy`; Dokploy services use Docker Provider (GHCR image pull). GitHub Actions builds/pushes images and triggers Dokploy deploy hooks.

  2) Service `talimy-web` (Docker Provider, image: `ghcr.io/<owner>/<repo>/web:<sha|latest>`, port: 3000; keep Dokploy Run Command empty if image CMD is correct).

  3) Domains (`talimy-web`): `talimy.space`, `www.talimy.space`, `*.talimy.space` (school subdomains). If platform is split, map `platform.talimy.space` to a dedicated service.

  4) Service `talimy-platform` (optional split; Docker Provider, image: `ghcr.io/<owner>/<repo>/platform:<sha|latest>`, domain: `platform.talimy.space`, env same scope as web).

  5) Service `talimy-api` (Docker Provider, image: `ghcr.io/<owner>/<repo>/api:<sha|latest>`, domain: `api.talimy.space`, `QUEUE_WORKERS_ENABLED=false`).

  6) Service `talimy-worker` (BullMQ worker; usually reuses `api` image, no domain, Run Command: `bun run --cwd apps/api start:worker:prod`, `QUEUE_WORKERS_ENABLED=true`).

  7) Web env scope: `apps/web/.env.local` - only `NEXT_PUBLIC_*` vars (+ `NODE_ENV`, `PORT` in Dokploy service env).

  8) API env scope: `apps/api/.env` - `DATABASE_URL`, `REDIS_URL`, `JWT_*`, `R2_*`, `RESEND_*`, `TWILIO_*`, `OPENROUTER_*`/`ANTHROPIC_API_KEY`, `SENTRY_DSN`, `AXIOM_TOKEN`, `PERMIFY_*`.

  9) SSL/Proxy + Health checks: wildcard/API/platform certs; web `/`, api `/health`, worker queue-processing smoke; verify subdomain routing.

  10) Rollback: previous successful Dokploy release + GHCR sha-tag re-deploy + DNS/SSL re-check.

2.3 Folder Naming Rules

•       Lowercase: "Alibek Karimov" → "alibek-karimov"

•       Replace spaces: " " → "-"

•       Remove special chars: o' → o, g' → g

•       Conflict prevention: timestamp suffix if needed

TypeScript kodi:

 

function generatePath(school: string, role: string, group: string, name: string): string {

  const clean = (s: string) => s.toLowerCase()

    .replace(/['\s]+/g, "-")

    .replace(/[^a-z0-9-]/g, "");

  return `/${clean(school)}/${role}/${clean(group)}/${clean(name)}/`;

}

3.2 XSS Prevention

Cross-Site Scripting himoyasi:

  1. React: Default XSS protection (escaped output)

  2. DOMPurify: Rich text sanitization

  3. CSP Headers: script-src 'self'

  4. Input validation: Zod schema

 

 

import DOMPurify from "isomorphic-dompurify";

const clean = DOMPurify.sanitize(userInput, {

  ALLOWED_TAGS: ["b", "i", "u", "p", "br"],

  ALLOWED_ATTR: []

});

 

3.3 CSRF Protection

Cross-Site Request Forgery himoyasi:

  1. NextAuth CSRF token (automatic)

  2. SameSite=Strict cookies

  3. Double submit cookie pattern

  4. Origin/Referer header validation

 

3.4 SQL Injection Prevention

•       Drizzle ORM: Parameterized queries only

•       No raw SQL without .sql template tag

•       Input validation: Zod schema

 

// SAFE ✅

await db.select().from(students).where(eq(students.id, studentId));

// UNSAFE ❌ (never do this)

// await db.execute(`SELECT * FROM students WHERE id = ${id}`);

Validation Architecture Note (2026-02-25): Backend REST request contracts use shared Zod schemas as the single source of truth. DTO classes should be derived from the same schemas (Hybrid pattern, e.g. `nestjs-zod` / `createZodDto`) to prevent runtime drift between global `ValidationPipe` (`class-validator`) and route-level Zod validation. `Teachers Module (2.6)` is the reference migration target for this pattern, and rollout priority before/through `2.6` is: Auth (`2.3`) -> Tenants (`2.4`) -> Users (`2.5`) -> Teachers (`2.6`).
